Okay, folks, let’s talk about this. This isn’t my first article about evaluating AI agents (and AI products in general), but it’s going to be the most comprehensive and in-depth one I’ve written. First things first — if you haven’t figured out by now that the magic behind great AI products like Lovable, Cursor, and the rest isn’t a smarter model but better evaluations, go stand in the corner. This is the industry’s open secret. Evaluation is the name of the game because we’re playing a statistical game. And I’m not the only one saying this: 👈 Hamel Husain , who has trained product managers and engineers from companies like OpenAI, Anthropic, and Google, wrote that unsuccessful AI products almost always share a common root cause: a failure to build robust evaluation systems. He and his partner, Shreya Shankar, report that in projects they’ve worked on, 60–80% of development time was spent on error analysis and evaluation. Yes, you read that right… the vast majority of the time. 👈 Anthropic (the company behind our beloved Claude) recommends an eval-driven development approach: build evals that define planned capabilities before the agent can fulfill them, then iterate until performance is satisfactory. They write that teams without evals get stuck in a reactive loop — catching issues only in production, where every fix creates new problems. 👈 Simon Willison (creator of Django) wrote that a robust approach to evals is “the single most important distinguishing factor between well-engineered, reliable AI systems and YOLO-cross-your-fingers-and-hope-it-works development.” Long story short, I hope you’re convinced. If not, go watch the video I posted on Twitter that illustrates how evals are the solution to all your problems in life. By the way, I’ve previously written about evals in the context of the PM tool stack in the AI era, and about the evaluation layer as the make-or-break of every AI product. This article takes it a step further, delving into the technical and practical aspects. So say hello to a concept called eval-driven development , or as I like to call it (because it’s really just a new name for an ancient concept): “test-driven development, but we’re in a stochastic world, so we needed a fancy new name.” Let’s dive in. Paradigm Shift: From Deterministic Testing to the LLM World In the old world, we had unit tests (testing individual components) and integration tests (testing how components work together). You gave it input X, you expected output Y. Simple. Deterministic. 1+1=2, every time. In the world of LLMs, everything changed: ◾ The non-deterministic nature of models: the same input can produce different outputs on every run ◾ The focus shifts to testing the application/agent’s ability to respond to specific user tasks ◾ You need to examine output quality: relevance, coherence, accuracy (and of course, making sure it doesn’t do things you really don’t want it to do, like, say, cursing at the user). And also — whether the path to the answer is optimal. In short, we’re no longer asking “is the answer identical to the expected answer?” but rather “is the answer good enough ?” And here, dear product managers, you need to define what “good enough” means (more on that in the next article). Traces, or: The Logs of the New World Now for the serious problem. We’ve established that we need to test every step in our agent’s operation, which is no simple task. You know that frustrating moment when your favorite developer explains there aren’t enough logs to investigate the bug? So you need a logging release befo re you can understand what happened? Welcome to a world where the new logs are called traces , and chances are you don’t have enough of them either — or at the level needed to understand why your beloved AI product just gave a completely delusional response. Traces are the observability layer for agents. Without good traces (and invest in understanding what that means), you’re blind. You know something isn’t working, but you have no idea what or where. Three Techniques for Running Evaluations Let’s assume your observability layer is excellent and you have good coverage. But that layer is just the foundation on top of which you build your evaluations. So how do you actually run evals? There are three main approaches: ◾ Code-Based Evals: Running code that compares outputs to expected results, performs calculations on outputs, etc. This is the most deterministic approach — useful when you have clear, measurable criteria. ◾ LLM-as-a-Judge: Using a separate LLM to evaluate your application’s outputs. Yes, it sounds insane — using AI to judge AI — but it’s one of the most recommended techniques. Keep in mind, of course, that it can never be 100% accurate, because well… it relies on AI. Important tip: always use discrete labels — “correct” vs. “incorrect,” not a scale of 1%–100%. Why? Because LLMs are terrible at continuous ratings. They’re much better at binary decisions. ◾ Human Annotations: Using human judgment (annotators or user feedback) to evaluate outputs. The most accurate approach, but also the most expensive and slow. And of course, you can combine multiple approaches :) Now that we have the techniques, let’s understand what we actually apply them to. What Do You Evaluate in an Agent? Three Dimensions Great, so we’ve got well-instrumented traces and understand which evaluation methods exist, but how do we decide what to actually evaluate? Let’s look at the agent’s components: ◾ The Router Essentially, the agent’s first call — and therefore the most critical one. We need to check two things here: Did the agent choose the right tool? And did it extract the correct parameters from the question? Even if the right tool was chosen, wrong parameters = worthless result. Important to understand: a router failure is a cascading failure — it feels like a failure of the final output, but the root cause is already in the first step. ◾ The Skills / Functions What the tool itself does. Here, you can use standard LLM evaluations, and the interesting point is that every tool in your agent will have completely different criteria. A tool that generates SQL? Check that the SQL runs and returns correct results (code-based). A tool that analyzes data? Check that the analysis is clear and the entities are correct (LLM-as-a-Judge). A tool that generates code? Check that the code actually runs without exceptions (code-based). ◾ The Path — And the Hardest of All The router chose correctly. Every tool did its job. But — did the agent take the right path? This is the biggest challenge for scalable evaluation, because the path can change between runs. Same question, same agent, and suddenly it takes a completely different route. Remember? Stochastic. And that’s exactly what brings us to the next concept — Convergence. Convergence: The Metric That Should Be on Your Dashboard Convergence measures how closely your agent follows the optimal path for a given query. Think of it this way — if there’s an optimal path of 5 steps and your agent does it in 5 steps, the convergence score is 1. If it does it in 10 steps, the score is 0.5. The formula: Overall Convergence Score = Σ min(1, S_optimal / S_agent,i) / N Meaning, for each query, you check the ratio between the optimal number of steps and the number of steps the agent actually took, capping at 1 (because if the agent took fewer steps than optimal, you still don’t give more than 1). Why does this matter in practice? Because when the convergence score drops, it means your agent is starting to waste steps, tokens, and money. This is the metric that tells you “something changed, investigate” long before users start complaining. Okay, But What Does This Actually Look Like? — Observability Tools for Agents Up until now, we’ve been talking theory. Now let’s talk about the tools you actually use to get your hands dirty.
